# 数据库升级指南

本项目使用 Alembic 管理业务数据库结构。SQLAlchemy 模型仍定义在 `app/db/models.py`，但表结构创建和后续变更必须通过 Alembic revision 落库。

## 基本原则

- 修改 `app/db/models.py` 后，必须同步生成并提交 Alembic 迁移文件。
- 当前项目没有旧数据处理需求，除非需求明确要求，不编写历史数据回填、修复或多版本兼容脚本。
- `DATABASE_AUTO_CREATE_DATABASE=true` 只负责在 PostgreSQL 数据库不存在时尝试建库。
- `DATABASE_AUTO_CREATE_TABLES=true` 负责后端启动时自动执行 `alembic upgrade head`，不再使用 `metadata.create_all` 管理结构。
- 生产多实例部署建议先由 CI/CD 或单独运维步骤执行升级，再启动或扩容后端实例，避免多个实例同时抢占迁移流程。

## 常用命令

所有命令默认通过根目录 `.env` 读取数据库连接。

```powershell
make db-current
make db-history
make db-upgrade
make db-downgrade
make db-revision MSG="添加用户字段"
make db-reset
```

命令说明：

- `make db-upgrade`：升级到最新版本，等价于 `alembic upgrade head`。
- `make db-upgrade DB_TARGET=202605170001`：升级或前进到指定版本。
- `make db-downgrade`：默认回滚一个版本，等价于 `alembic downgrade -1`。
- `make db-downgrade DB_TARGET=base`：回滚到基线之前，通常只用于本地验证或清库。
- `make db-revision MSG="描述"`：根据当前模型和数据库差异生成迁移文件。
- `make db-current`：查看当前数据库版本。
- `make db-history`：查看迁移历史。
- `make db-reset`：删除并重建开发数据库，再执行迁移并初始化管理员账号。

## 开发流程

1. 修改 `app/db/models.py` 中的 SQLAlchemy 模型。
2. 确认本地 `.env` 指向可用于生成迁移的开发数据库。
3. 执行：

```powershell
make db-revision MSG="描述本次结构变更"
```

4. 打开新生成的 `alembic/versions/*.py`，检查表、列、索引、唯一约束、外键和 `ondelete` 是否符合预期。
5. 执行：

```powershell
make db-upgrade
make db-current
```

6. 运行相关测试。涉及数据库结构时，至少运行：

```powershell
uv run --env-file .env pytest tests/test_db_init.py tests/test_db_migrations.py
```

## 本地重建数据库

本地数据不需要保留时，可以直接重建开发库：

```powershell
make db-reset
```

默认管理员账号由 Makefile 传入：

```powershell
make db-reset ADMIN_EMAIL=admin@example.com ADMIN_PASSWORD=StrongPass123
```

`db-reset` 会先删除并重建数据库，再执行 Alembic 升级到最新版本，最后初始化管理员账号。

## 启动时自动升级

后端启动时会读取 `DATABASE_AUTO_CREATE_TABLES`：

- `true`：启动时自动执行 `alembic upgrade head`。
- `false`：启动时不改动数据库结构，需要部署流程提前执行 `make db-upgrade` 或等价命令。

本地开发可以保留默认值 `true`。生产或多实例环境建议设为 `false`，把升级放到发布流水线中的单独步骤。

## 部署建议

推荐发布顺序：

1. 部署前备份数据库。
2. 使用新代码镜像或新代码目录执行 `make db-upgrade`。
3. 确认 `make db-current` 输出为最新版本。
4. 启动或滚动更新后端服务。
5. 运行健康检查和关键接口验证。

如果发布失败且迁移包含可逆 `downgrade`，可按迁移文件内容评估后执行：

```powershell
make db-downgrade DB_TARGET=上一个版本号
```

含有数据删除、列删除或不可逆数据转换的迁移，不应盲目回滚，必须先确认备份和业务影响。

## 迁移文件审查清单

- 是否只包含本次模型变更相关的 DDL。
- 是否保留了必要的索引和唯一约束。
- 外键的 `ondelete` 是否与模型关系一致。
- nullable、字段长度和类型是否符合接口和业务预期。
- 是否误生成了无关表、无关索引或临时实验字段。
- downgrade 是否至少能在本地开发库中正常执行。

## 测试说明

测试环境使用 SQLite 内存库，并通过 Alembic 迁移创建业务表。为了让测试保持快速，清理阶段仍会直接删除业务表和 Alembic 版本表；迁移专项测试负责覆盖从空库升级和回滚到 base 的行为。
